PokéTrade

Project & Security Brief

A location-aware Pokémon trading app: see trainers near you, see what they're offering, and set up a trade in a couple of taps.

👉 Click here to open the app View the website

1 · What it is

The pitch: with the current Pokémon craze, people want to trade — but finding someone nearby who has what you want (and wants what you have) is word-of-mouth and luck. PokéTrade puts trainers on a map, shows each person's offers and wants, and matches trades locally.

🗺️ Nearby map

See trainers around you with approximate distance — never an exact address.

🔁 Trade board

Every trainer lists what they offer and what they want. Propose a trade in one tap.

💬 Trade requests

A proposal opens a chat thread to agree on the swap and a meeting spot.

⭐ Reputation

Ratings after each completed trade keep the community trustworthy.

2 · What you're looking at today

The link above is a working skeleton — the full user experience with sample data, so you can judge the flow and look before we invest in the backend. It's deliberately lightweight: plain HTML/CSS/JS, no frameworks, loads instantly on any phone.

3 · How we'll make it work

PiecePlan
Accounts Supabase Auth (email code or Google/Apple sign-in). Every trainer gets a profile: display name, avatar, offers/wants lists.
Location Browser geolocation (opt-in) → we store a rounded grid cell, not exact coordinates. The map (Leaflet + OpenStreetMap) shows trainers by area.
Database Supabase Postgres: profiles, listings (offers/wants), trades, messages, ratings. We already run this stack in production for the Lockdown Lab registration site.
Live updates Supabase Realtime pushes new nearby trades and chat messages instantly.
Trade flow Propose → accept/decline → chat → mark completed → both sides rate each other.
Distribution Progressive Web App: works from a link on any phone (as it does now), installable to the home screen — no app-store gatekeeping to launch. Native wrappers can come later.

4 · Security & privacy

A location app lives or dies on trust, so this is designed in from day one — the same model already protecting our registration platform.

Location privacy (the big one)

  • Approximate by design: we never store or show exact coordinates — the server keeps only a rounded grid cell (~500 m). "0.4 km away", never a pin on a house.
  • Opt-in and reversible: location sharing is off until the user turns it on, and can be paused (ghost mode) any time.
  • Fresh, not historical: only the latest area is kept — no movement history, no location trail to leak.

Data protection

  • Row-Level Security on every table: the database itself enforces who can read what. Anonymous visitors can read nothing personal; users see only what's meant to be public (name, offers/wants, approximate area) — never emails or contact details.
  • No secrets in the browser: the client uses only a publishable key that RLS constrains. Privileged operations run server-side in edge functions holding the service-role key — the same split our current admin dashboard uses.
  • HTTPS everywhere, minimal data collected, delete-account = data removed.

People safety

  • In-app contact only: chat opens after a trade is accepted — no phone numbers or emails exposed.
  • Report & block on every profile; blocked users vanish from your map.
  • Reputation: ratings and trade history make bad actors visible fast.
  • Safe-meet guidance: the app suggests public meeting spots when a trade is agreed.
  • Age gate at sign-up; community guidelines enforced by the report flow.

Abuse prevention

  • Rate limiting on sign-ups, proposals and messages (edge functions).
  • Server-side validation of every write — the client is never trusted.
  • Audit trail on trades and reports for dispute resolution.

5 · Rollout

  • Phase 1 (now): skeleton — validate the concept and the flow. ✅
  • Phase 2: accounts, real profiles, offers/wants, trade proposals.
  • Phase 3: real map + approximate location, chat, ratings.
  • Phase 4: polish, installable PWA, invite-based local launch.
